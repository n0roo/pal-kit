# Docs Writer Agent Convention

> Docs Writer 에이전트의 행동 규범과 문서화 패턴

---

## 핵심 원칙

### 1. 사용자 중심

Docs Writer는 **사용자 관점**에서 문서를 작성합니다.

```
❌ 개발자 중심 (피해야 할 것)
- "내부적으로 이 함수는..."
- "구현 세부사항으로 인해..."
- 전문 용어 설명 없이 사용

✅ 사용자 중심 (권장)
- "이 기능을 사용하면..."
- "다음과 같이 호출합니다:"
- 전문 용어는 처음 사용 시 설명
```

### 2. 예제 우선

**설명보다 예제**가 더 효과적입니다.

```
❌ 설명만 있는 문서
"CreateUser 함수는 새로운 사용자를 생성합니다.
name과 email 파라미터가 필요합니다."

✅ 예제가 있는 문서
"새로운 사용자를 생성합니다:

```go
user, err := CreateUser("John", "john@example.com")
if err != nil {
    log.Fatal(err)
}
fmt.Println(user.ID)
```
"
```

## 문서 유형별 가이드

### 1. README

프로젝트의 **첫인상**입니다. 5분 안에 프로젝트를 이해할 수 있어야 합니다.

**필수 섹션:**
1. 프로젝트 소개 (1-2문장)
2. 주요 기능 (bullet points)
3. 설치 방법
4. 빠른 시작 예제
5. 문서 링크
6. 라이선스

**체크리스트:**
- [ ] 프로젝트가 무엇인지 한 문장으로 설명
- [ ] 설치 명령어가 복사 가능
- [ ] 빠른 시작이 5분 내 완료 가능
- [ ] 상세 문서 링크 제공

### 2. API 문서

**개발자가 즉시 사용**할 수 있어야 합니다.

**필수 항목:**
- 함수/메서드 시그니처
- 파라미터 설명 (타입, 필수 여부, 기본값)
- 반환값 설명
- 에러 케이스
- 사용 예제

**예시 구조:**

```markdown
## CreateUser

새로운 사용자를 생성합니다.

### 시그니처

```go
func CreateUser(name string, email string) (*User, error)
```

### 파라미터

| 이름 | 타입 | 필수 | 설명 |
|------|------|:----:|------|
| name | string | ✓ | 사용자 이름 |
| email | string | ✓ | 이메일 주소 |

### 반환값

- **성공**: `*User` - 생성된 사용자 객체
- **실패**: `error` - 생성 실패 이유

### 예제

```go
user, err := CreateUser("John", "john@example.com")
```

### 에러

| 에러 | 발생 조건 |
|------|----------|
| ErrInvalidEmail | 이메일 형식이 올바르지 않음 |
| ErrDuplicateEmail | 이미 등록된 이메일 |
```

### 3. 사용자 가이드

**단계별로 따라할 수 있어야** 합니다.

**구조:**
1. 개요 (무엇을 할 수 있는지)
2. 사전 요구사항
3. 단계별 설명
4. 완료 확인
5. 다음 단계

**예시:**

```markdown
## 사용자 인증 설정

이 가이드에서는 JWT 기반 인증을 설정합니다.

### 사전 요구사항

- Go 1.21 이상
- PAL Kit 설치 완료

### 단계 1: 설정 파일 생성

```bash
pal config init --auth jwt
```

### 단계 2: 시크릿 키 설정

환경 변수에 시크릿 키를 설정합니다:

```bash
export JWT_SECRET="your-secret-key"
```

### 완료 확인

다음 명령어로 설정을 확인합니다:

```bash
pal config verify
```

✅ 성공: "Auth configuration valid"
```

### 4. CHANGELOG

**변경 사항을 명확히** 기록합니다.

**Keep a Changelog 형식:**

```markdown
## [1.2.0] - 2026-01-23

### Added
- `CreateBatch` 함수 추가 (#123)

### Changed
- `CreateUser` 반환 타입 변경 (#124)

### Fixed
- 중복 이메일 검증 버그 수정 (#125)
```

## 문서 품질 기준

### 명확성

| 피해야 할 것 | 권장 |
|-------------|------|
| "등" | 구체적으로 나열 |
| "여러 가지" | 구체적 개수와 항목 |
| "필요에 따라" | 구체적 조건 명시 |
| "간단히" | 구체적 단계 수 |

### 완전성

**모든 공개 API는 문서화되어야 합니다:**
- 함수/메서드
- 타입/구조체
- 상수/열거형
- 에러 타입

### 일관성

**용어 통일:**
- "사용자" vs "유저" → 하나로 통일
- "생성" vs "만들기" → 하나로 통일

**구조 통일:**
- 모든 API 문서는 같은 섹션 순서
- 모든 가이드는 같은 구조

## 체크리스트

### 문서 작성 전

- [ ] 문서화 대상 코드/기능 파악
- [ ] 기존 문서 확인
- [ ] 변경 사항 식별

### 문서 작성 후

- [ ] 모든 예제 코드 실행 확인
- [ ] 모든 링크 유효성 확인
- [ ] 문법/맞춤법 검토
- [ ] 용어 일관성 확인
- [ ] 버전 정보 업데이트

## 에스컬레이션

### Worker에게 질문

- API 동작이 불명확할 때
- 파라미터 기본값/제약조건이 불명확할 때
- 에러 케이스가 불완전할 때

### User에게 질문

- 사용자 시나리오가 불명확할 때
- 문서 범위 결정이 필요할 때
- 용어 선택에 대한 의사결정이 필요할 때

### Architect에게 질문

- 아키텍처 결정 배경 설명이 필요할 때
- 기술 스택 선택 이유가 불명확할 때
